# Purpose

Default rules for compiled markdown content wikis used with **agwiki** `ingest`. Paths in prose below are relative to the wiki root; the substituted `{{WIKI_ROOT}}` value is an absolute path for the agent.

# Source of truth

Agents MUST write entity files to `content/<kind>/`. Do not write entity pages directly to `wiki/`; `wiki/` is generated by `agwiki compile`.

# Front matter

Every entity file must start with YAML front matter:

```yaml
---
id: "<ulid>"
title: "Entity Title"
schema_version: 1
---
```

Required fields: `id`, `title`, and `schema_version: 1`.
Optional fields: `status` (`active` or `archived`), `relations` (`target` id plus `rel` string), and `aliases`.

# Rules

- Never modify files inside `raw/`
- Treat `raw/` as source material
- Create and update entity files only inside `content/<kind>/`
- Prefer updating existing entities instead of creating duplicates
- Preserve uncertainty and disagreements between sources
- Use concise markdown
- Add relations between related entity ids when useful
- Keep `content/pages/index.md` updated
- Append important actions to `content/pages/log.md`

# Folder meanings

- `raw/` — original source documents (optional subfolders such as notes, articles, tweets; paths are stable identifiers)
- `content/sources/` — source summary entity files
- `content/concepts/` — concept entity files
- `content/projects/` — project entity files
- `content/people/` — person entity files
- `content/topics/` — broader topic entity files
- `content/syntheses/` — higher-level synthesis entity files
- `content/pages/` — non-entity navigation pages such as `index.md`, `inbox.md`, and `log.md`
- `wiki/` — generated output; read it when answering questions, but do not edit it directly

# Ingest workflow

When asked to ingest a file from `raw/`:

1. Read the source
2. Create or update source summary entities in `content/sources/`
3. Create or update related concept/topic/project/person/synthesis entities
4. Ensure every entity has valid YAML front matter with `id`, `title`, and `schema_version: 1`
5. Add `relations` entries between related ids where useful
6. Update `content/pages/index.md` when navigation or top-level structure changes
7. Append a short note to `content/pages/log.md`

# Query workflow

When asked a question:

1. Start from `content/pages/index.md` or `wiki/index.md`
2. Read relevant content entity files or generated wiki pages
3. Answer using the wiki
4. If useful, create or improve reusable source entities under `content/<kind>/`

# Maintenance workflow

When asked to clean or lint the wiki:

- find duplicates
- find contradictions
- find orphan pages
- find stale pages
- suggest missing pages

Use concise markdown; prefer updating existing pages over duplicates; preserve uncertainty between sources.
